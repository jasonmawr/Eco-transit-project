# Release Scope Definition — 100% of Current Brief

## Required for “complete according to current customer brief”

A release can be called complete only when these journeys pass UAT:

1. Guest searches a route from current location to a station/place and sees bus+metro options.
2. Guest sees route adjusted/explained by rainy or hot weather context.
3. Guest opens a station page and discovers nearby places and guides.
4. Guest reads article/video guide for combining metro + bus.
5. User logs in and uploads a ticket image.
6. Moderator reviews low-confidence ticket.
7. Verified ticket awards points through ledger.
8. User redeems a voucher with points.
9. User submits a review/guide; moderator approves it; it appears publicly.
10. User creates a time bill and shares/copies it with privacy masking.
11. Admin can manage core content, vouchers, moderation, stations/POIs and audit.
12. Mobile 390px screenshots show no broken/ugly UI.

## Explicitly not required unless separately contracted

- Official ticket selling/payment processing.
- Real-time operator-grade ETA if official API is unavailable.
- Unlimited anti-fraud accuracy for all possible ticket types.
- Production legal approval by the development agent.
- Partner voucher API if no partner contract exists.

## Required honesty in UI

If a route, weather, fare, voucher or place is estimated or sourced from seed data, UI must state that gently in Vietnamese.
